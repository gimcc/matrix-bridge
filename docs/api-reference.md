# Matrix Bridge API Reference

Base URL: `http://<bridge-host>:29320`

All Bridge API request and response bodies are JSON unless noted otherwise.

## Authentication

Bridge HTTP routes under `/api/v1/*` are optionally protected by `appservice.api_key`.

When configured, send:

```http
Authorization: Bearer <api_key>
```

Notes:

- Bridge HTTP API accepts header auth only.
- WebSocket authentication is separate and happens through the first WS frame.
- `hs_token` is only for Synapse <-> appservice traffic on `/_matrix/app/v1/*` routes.

## Rate Limiting

HTTP Bridge API routes are rate limited per client IP:

- `120` requests per second
- burst size `300`

The limiter does not apply to:

- `/health`
- `/_matrix/app/v1/*`
- `/api/v1/ws`

## Send Inbound Message

```http
POST /api/v1/message
```

Use this when an external platform wants to inject a message into a mapped Matrix room.

### Request body

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `platform` | string | Yes | Platform ID, 1..64 chars, alphanumeric / `_` / `-` / `.` |
| `room_id` | string | Yes | External room ID, not Matrix room ID |
| `sender.id` | string | Yes | External sender ID |
| `sender.display_name` | string | No | Puppet display name |
| `sender.avatar_url` | string | No | External avatar URL |
| `content` | object | Yes | See [Inbound Content Types](#inbound-content-types) |
| `external_message_id` | string | No | Deduplication key; autogenerated if omitted |
| `reply_to` | string | No | External message ID being replied to |

Example:

```json
{
  "platform": "telegram",
  "room_id": "chat_123",
  "sender": {
    "id": "user_42",
    "display_name": "Alice"
  },
  "content": {
    "type": "text",
    "body": "Hello",
    "html": "<b>Hello</b>"
  },
  "external_message_id": "msg-001"
}
```

Response:

```json
{
  "event_id": "$event:example.com",
  "message_id": "msg-001"
}
```

Notes:

- `room_id` and `sender.id` are sanitized before they are used in storage or puppet MXIDs.
- If the `(platform, external_message_id)` pair already exists, the bridge treats it as a duplicate.
- `reply_to` and edit / reaction / redaction targets refer to external message IDs.

## Upload Media

```http
POST /api/v1/upload
Content-Type: multipart/form-data
```

Form fields:

- `file`: required, first multipart file field is used

Response:

```json
{
  "content_uri": "mxc://example.com/abc123",
  "filename": "photo.png",
  "size": 12345
}
```

Notes:

- Maximum size is `200 MiB`.
- The upload is forwarded to the Matrix media repository.
- Returned `content_uri` can be reused in `image`, `file`, `video`, or `audio` payloads.

## Room Mappings

### Create or reuse a mapping

```http
POST /api/v1/rooms
```

Request body:

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `platform` | string | Yes | External platform ID |
| `external_room_id` | string | Yes | External room identifier |
| `matrix_room_id` | string | No | Existing Matrix room ID; omit to auto-create |
| `room_name` | string | No | Used only when auto-creating a Matrix room |
| `invite` | array<string> | No | Extra Matrix users to invite; ignored unless `allow_api_invite = true` |

Example:

```json
{
  "platform": "slack",
  "external_room_id": "C123456",
  "room_name": "Slack / General",
  "invite": ["@ops:example.com"]
}
```

Behavior:

- Returns `201` when a new mapping is created.
- Returns `200` when the mapping already exists.
- When `matrix_room_id` is omitted, the bridge creates a new Matrix room.
- Auto-created rooms always include `appservice.auto_invite`.
- When `allow_api_invite = true`, request-level `invite` entries are merged into `auto_invite`.
- Newly created mappings are also attached to the platform's Matrix Space when spaces are available.

Response:

```json
{
  "id": 1,
  "matrix_room_id": "!room:example.com"
}
```

### Delete a mapping

```http
DELETE /api/v1/rooms/{id}
```

Response:

```json
{ "deleted": true }
```

### List mappings

```http
GET /api/v1/admin/rooms?platform=telegram&after=0&limit=100
```

Query parameters:

- `platform`: optional filter
- `after`: optional cursor, defaults to `0`
- `limit`: optional page size, clamped to `1..1000`

Response:

```json
{
  "rooms": [
    {
      "id": 1,
      "matrix_room_id": "!room:example.com",
      "platform_id": "telegram",
      "external_room_id": "chat_123",
      "created_at": "2026-03-27 00:00:00"
    }
  ],
  "next_cursor": 1
}
```

## Webhooks

### Register or update a webhook

```http
POST /api/v1/webhooks
```

Request body:

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `platform` | string | Yes | Target platform |
| `url` | string | Yes | `http` or `https` webhook URL |
| `events` | string | No | Stored with the webhook, defaults to `message`; current delivery logic does not hard-filter outbound events by this field |
| `forward_sources` | string or array<string> | No | Allowlist for non-Matrix source platforms |
| `capabilities` | string or array<string> | No | Supported features advertised by this integration |
| `owner` | string | No | Matrix user auto-invited into portal rooms for this platform |

Example:

```json
{
  "platform": "telegram",
  "url": "https://hooks.example.com/tg",
  "events": "message,redaction",
  "forward_sources": ["matrix", "slack"],
  "capabilities": ["message", "image", "command"],
  "owner": "@bridge-admin:example.com"
}
```

Semantics:

- Matrix-user messages are always eligible for forwarding.
- `forward_sources` controls forwarding for non-Matrix sources only.
- Empty `forward_sources` means "Matrix only".
- `"*"` means "allow all sources".
- Re-registering the same `(platform, url)` updates the stored settings and re-enables the webhook.

When `appservice.webhook_ssrf_protection = true`, webhook URLs targeting localhost, metadata endpoints, private IPs, loopback, link-local, CGNAT, and other reserved ranges are rejected.

Response:

```json
{ "id": 1 }
```

### Delete a webhook

```http
DELETE /api/v1/webhooks/{id}
```

Response:

```json
{ "deleted": true }
```

### List webhooks

```http
GET /api/v1/admin/webhooks?platform=telegram&after=0&limit=100
```

Response:

```json
{
  "webhooks": [
    {
      "id": 1,
      "platform_id": "telegram",
      "webhook_url": "https://hooks.example.com/tg",
      "events": "message,redaction",
      "enabled": true,
      "forward_sources": "matrix,slack",
      "capabilities": "message,image,command",
      "owner": "@bridge-admin:example.com"
    }
  ],
  "next_cursor": 1
}
```

## WebSocket

```http
GET /api/v1/ws?platform=telegram&forward_sources=*&capabilities=message,image,command
```

Query parameters:

| Parameter | Required | Notes |
|-----------|----------|-------|
| `platform` | Yes | Platform subscription key |
| `forward_sources` | No | Comma-separated source allowlist |
| `capabilities` | No | Comma-separated capability list |

When `api_key` is configured, the client must send the first frame within 10 seconds:

```json
{ "access_token": "<api_key>" }
```

Notes:

- Do not send API keys in the WebSocket URL.
- The bridge keeps up to `1000` WS clients across all platforms.
- Clients receive the same payload shapes used for webhooks.

### Message event

```json
{
  "event": "message",
  "platform": "telegram",
  "source_platform": "slack",
  "message": {
    "id": "$event:example.com",
    "sender": {
      "platform": "slack",
      "external_id": "alice",
      "display_name": "Alice",
      "avatar_url": "https://example.com/alice.png"
    },
    "room": {
      "platform": "telegram",
      "external_id": "chat_123",
      "name": null
    },
    "content": {
      "type": "text",
      "body": "Hello",
      "formatted_body": "<b>Hello</b>"
    },
    "timestamp": 1710000000000,
    "reply_to": null
  }
}
```

`source_platform` is present only for cross-platform forwarded messages.

```json
{
  "event": "command",
  "platform": "telegram",
  "sender": "@user:example.com",
  "command": "/start",
  "room_id": "!dm:example.com"
}
```

## Inbound Content Types

These shapes are accepted by `POST /api/v1/message`.

### Text

```json
{ "type": "text", "body": "Hello", "html": "<b>Hello</b>" }
```

### Notice / emote

```json
{ "type": "notice", "body": "Bridge notice" }
{ "type": "emote", "body": "waves" }
```

### Media

```json
{ "type": "image", "url": "mxc://example.com/abc", "caption": "Photo", "mimetype": "image/png" }
{ "type": "file", "url": "mxc://example.com/abc", "filename": "doc.pdf", "mimetype": "application/pdf" }
{ "type": "video", "url": "mxc://example.com/abc", "caption": "Clip", "mimetype": "video/mp4", "duration": 30 }
{ "type": "audio", "url": "mxc://example.com/abc", "mimetype": "audio/ogg", "duration": 5 }
```

### Location

```json
{ "type": "location", "latitude": 48.8566, "longitude": 2.3522 }
```

### Reaction / redaction / edit

```json
{ "type": "reaction", "target_id": "msg-001", "emoji": "👍" }
{ "type": "redaction", "target_id": "msg-001" }
{
  "type": "edit",
  "target_id": "msg-001",
  "new_content": { "type": "text", "body": "corrected text" }
}
```

## Outbound Content Notes

Webhook and WebSocket message payloads use the internal `MessageContent` model:

- text formatting is emitted as `formatted_body`, not `html`
- edits are emitted with `new_content` containing the replacement `MessageContent`
- reactions, edits, and redactions reference the external `target_id` (the platform message ID from the original message mapping, not the Matrix event ID)
- location coordinates are extracted from the Matrix `geo:` URI and emitted as `latitude`/`longitude` floats
- all content payloads keep the same `type` tag values as inbound requests
